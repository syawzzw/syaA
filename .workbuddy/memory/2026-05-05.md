### 性能优化 — 流式下载（第四轮）
- **实测 sudugu.org TXT直链速度**：id=2 p=1 单卷 3.6MB，耗时 308s（~5分钟），平均 11.8 KB/s，TTFB 2.7s
- **问题**：原 `get_url_txt_without_cookie()` 用 `requests.get().text` 一次性读全部内容到内存
  - 3.6MB 文件需持续连接 ~5 分钟，连接不稳定导致 IncompleteRead
- **修复**：新增 `get_url_txt_streaming(url, timeout=120)` 函数
  - 使用 `stream=True` + `iter_content(chunk_size=8192)` 分块读取
  - timeout 设为 `(30, 120)` 元组：connect=30s（足够TTFB），read=120s（两次chunk间最大等待）
  - 只要数据在持续传输就不会触发超时，比一次性读稳定得多
- **_download_txt_with_retry 改为调用** `get_url_txt_streaming()` 替代旧函数

### ⭐ 章节级增量更新（第七轮）
- **站点结构探索结果**：
  - 详情页：`/{id}/txt.html` → 元数据 + 分卷链接 + 最新章节号
  - 目录页：`/{id}/` (分页) → 全部章节列表（URL: `/{id}/{文章ID}.html`）
  - 单章页：`/{id}/{文章ID}.html` → ~1000字纯文本/章
  - TXT直链：`/txt/?id={id}&p={卷号}` → 整卷TXT（~3.6MB, ~12KB/s）
  - URL中的文章ID不连续（第12章=/2/28.html），不能简单递推
  - 例：id=2 有1449章，4个分卷，目录分2页

- **新增4个方法**：
  - `_extract_latest_chapter_num(html, book_id)` — 从详情页提取最新章节号
  - `_fetch_directory_chapters(base_url, book_id, known_max)` — 爬取目录页获取新章节列表（支持增量过滤）
  - `_download_single_chapter(chap_url, book_id, chap_label)` — 下载单章纯文本正文（多模式匹配+兜底清理）
  - `_fetch_book_meta` 返回值新增第4项：`latest_chapter_num`

- **混合增量策略（双层检测）**：
  ```
  增量更新流程:
  ① 拉详情页 → latest_chapter_num（如1450） vs 索引中（如1449）
  ② 相同→秒跳过 | 更大→进入③④
  ③ 分卷级：有新分卷？→ TXT直链下载（快）
  ④ 章节级：旧分卷内有新章？→ 逐章下载追加到文件末尾（~1KB/章，秒级）
  ⑤ 更新索引 latest_chapter_num
  ```

- **"已是最新"判断升级**：原来只看分卷号，现在同时检查章节号。即使4卷全有，只要章节号变大就自动追更
- **边界情况处理**：downloaded_parts为空但有新章节时，单独走章节追加分支

### 网络层重构 + SSL 错误自动恢复（第十三轮）
- **根因**：`requests.get()` 全局连接池复用陈旧 SSL 连接
  - 长时间下载（卷2: 80s, 1070KB）后底层 TCP/SSL 被服务端关闭
  - 后续请求复用池中已死连接 → `SSLEOFError` / `IncompleteRead`
  - 重试继续用同一个坏连接 → 3次全部秒败（~150ms）
- **核心改动**：
  1. **模块级 Session** (`_http_session`) 替代 `requests.get()` 全局连接池
     - 可主动清理连接池 `_cleanup_stale_connections()`
     - 统一配置 pool_connections=10, pool_maxsize=10, max_retries=0
  2. **错误分类函数** `_classify_error(e)` → 返回 `(分类名, 是否SSL/连接级)`
     - 区分：SSL连接错误 / IncompleteRead / 连接断开 / 超时 / HTTP状态码 / 其他
  3. **SSL 风暴检测** `_get_cooldown_seconds()`
     - 连续 SSL 错误计数器（60s 无错误则重置）
     - 第3次起每次额外冷却 +3s（3→6→9...上限15s）
  4. **所有请求路径统一更新**：
     - `get_url_txt_without_cookie()` — 内置重试+清理+冷却
     - `_download_txt_with_retry()` — 分卷下载，增强日志+冷却标签
     - `_fetch_book_meta()` — 详情页，增加错误分类标签
     - `_download_single_chapter()` — 单章下载，增加分类+清理
     - Phase 1 列表页采集，增加分类+清理
- **新增 import**：`import requests.adapters as adapters`

### 反爬防御 + 自适应降速（第十五轮）
- **触发原因**：Phase 1 全站采集 409 页，每页仅 0.5s 间隔 → 触发 sudugu.org 反爬
  - 表现：第25页起请求被重定向到 google.com（Cloudflare/类似防护）
  - 日志特征：`ConnectTimeoutError(www.google.com)` — 说明响应体是 google 页面而非目标内容
- **核心改动**：
  1. **基础延迟加大 + 随机化**
     - `time.sleep(0.5)` → `2.5s ± [0.8~1.5]s` 随机（避免固定间隔被识别为机器人）
  2. **反爬检测**（双层）
     - 层1：`get_url_txt_without_cookie()` 检查 `resp.url` 重定向链
       - 最终 URL 包含 google.com / challenges.cloudflare.com → 抛出 `AntiCaptchaRedirect`
     - 层2：Phase 1 循环检查页面内容特征
       - google/captcha/cloudflare/人机检测/CF挑战页 关键词命中
       - 正常列表页应含 `/txt.html` 链接，不含则标记可疑
     - 检测到反爬 → 清理连接池 + 冷却 10~60s（连续次数 ×10，上限60）+ 跳过本页
  3. **指数退避自适应延迟**（连续失败时）
     - `_consecutive_fails` 计数器
     - 失败1次：+2.5s 延迟 | 失败2次：+5s | ... | 上限30s
     - 成功一次立即重置为 0
  4. **人类模拟休息**：每采集50页额外暂停 5 秒
  5. **进度日志增强**：显示平均速度（s/页）+ 连续失败警告
- **新增异常类**：`AntiCaptchaRedirect(Exception)`
- **`_classify_error()` 新增识别**：`AntiCaptchaRedirect` → `'反爬重定向(验证码/Google)'`
- **新增 import**：`import random`
- **预计效果**：409 页约需 25~35 分钟（原 3.4 分钟），但不会再被封
